## 为什么需要这个改动

### 项目背景
公司主营 VS Code 插件开发,日志面板需要实现:
- 实时接收后端通过 Windows pipe 通道推送的日志数据
- 300ms 节流聚合数据后渲染(不做实时传输,降低页面压力)
- 数据双端处理:写入 C 盘本地文件持久化 + 前端页面渲染
- 页面最多存储 20000 条日志,超出后执行"首删尾插"
- 支持基于全量 20000 条数据的筛选和导出

### 核心问题
日志功能连续运行一整夜后出现页面白屏崩溃。通过 Performance 工具定位到:
1. **GC 频繁回收不及时**: 大量临时对象创建,垃圾回收超负荷,内存无法释放
2. **JS 主线程运行过于频繁**: 高频数据更新、渲染、滚动操作持续占用主线程
3. **内存持续上涨**: 最终引发内存溢出,导致页面崩溃

当前已做的优化包括:优化日志数组操作、虚拟列表渲染逻辑、滚动逻辑。但仍需确认是否能长期稳定运行。

## 改动内容

- **内存管理**: 实现高效的数据生命周期管理,包含适当的清理和有界的内存使用
- **数据处理**: 优化 `dealMessageData` 以减少对象创建和 GC 压力
- **渲染**: 改进 React 渲染效率以最小化 VDOM 垃圾
- **状态更新**: 使用不可变模式和对象池来减少内存分配
- **防抖**: 增强现有的滚动防抖,添加批量更新
- **垃圾回收**: 主动释放引用并在适用的地方使用 WeakRef
- **破坏性变更**: 更改内部数据结构(可能影响任何直接访问日志数据内部的代码)

## 能力

### 新增能力

- `log-memory-management`: 内存高效的日志存储,包含有界缓冲区、对象池和自动清理,防止内存泄漏
- `log-batch-processing`: 批量消息数据处理,减少 300ms 间隔更新带来的 GC 压力
- `log-rendering-optimization`: 优化的渲染,包含记忆化、稳定引用和减少 VDOM 垃圾

### 修改的能力

无 - 这是日志监控组件的新功能集。

## 影响

- **受影响的代码**: `src/webview/virtuoso.tsx` (React + React-Virtualized 虚拟列表)
- **数据结构**: `logAllDataRef` (全量 20000 条数据用于筛选), `logData` (虚拟列表展示数据)
- **通信方式**: VS Code 扩展进程与 Panel 页面的 IPC 通信
- **性能目标**: 内存使用稳定在 ~50MB,GC 频率降低 70%,长时间运行无内存溢出
- **用户体验**: 消除白屏崩溃,保证长时间调试会话的稳定性
- **测试**: 需要连续运行测试(24小时)验证稳定性
