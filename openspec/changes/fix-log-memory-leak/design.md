## 背景

### 业务场景
VS Code 插件日志面板的核心功能:
- **数据源**: 后端通过 Windows pipe 通道实时推送日志
- **节流策略**: 300ms 节流聚合数据(不做实时渲染,降低压力)
- **双端处理**: 写入 C 盘本地文件持久化 + 前端页面渲染
- **数据管控**: 最多存储 20000 条,超出后"首删尾插"
- **交互功能**: 基于全量数据的筛选、导出保存

### 技术架构
- **前端框架**: React
- **虚拟列表**: React Virtualized 4.18.4 (大数据量高效渲染)
- **通信方式**: VS Code 扩展进程与 Panel 页面 IPC 通信
- **数据结构**: 
  - `logAllDataRef`: 全量 20000 条数据(用于筛选)
  - `logData`: 虚拟列表展示数据

### 性能问题分析
运行 Performance 工具排查发现的问题:

1. **数组展开导致内存碎片化**: `const allList = [...prev, ...processedDatalist]` 在每次更新时创建新数组
2. **紧密循环中的对象创建**: `dealMessageData` 每 300ms 为每个日志条目创建新对象
3. **状态更新抖动**: `setLogData` 触发重新渲染,即使数据可以被批量处理
4. **保留的引用**: `logAllDataRef` 即使在过滤时也将所有数据保留在内存中
5. **VDOM 垃圾**: React 为每次状态更新创建新的 fiber 节点

GC 每几秒运行一次,收集数兆字节的 JS 对象,导致主线程持续占用,最终内存溢出崩溃。

## 目标 / 非目标

**目标:**
- 减少 70% 以上的内存分配率
- 消除长时间使用(8小时以上)期间的 UI 冻结
- 保持响应式滚动和过滤
- 将 20k 日志条目的内存使用限制在约 50MB
- 支持连续日志记录过夜而不增长内存

**非目标:**
- 完全重写组件
- 更改 WebSocket 通信协议
- 修改 react-virtuoso 库
- 减少日志保留数量(保持最大 20k 行)

## 决策

### 1. 使用可变 Ref 和手动更新(替代 React State)

**决策**: 使用带有手动渲染触发器的 `useRef` 存储日志数据,而不是用于高频更新的 `useState`。

**理由**:
- React state 在每次更新时创建新的数组引用,导致 VDOM 协调开销
- `useRef` 允许原地修改而不触发重新渲染
- 我们控制何时将更新刷新到 UI(每 500ms 批量处理而不是每 300ms)

**考虑的替代方案**:
- **useReducer**: 仍然创建新的状态对象,GC 压力相同
- **外部状态 (Redux/Zustand)**: 增加复杂性,仍然存在相同的基本问题
- **Web Workers**: 对此用例来说过于复杂,增加消息传递开销

### 2. 实现循环缓冲区用于日志存储

**决策**: 使用循环缓冲区(环形缓冲区)存储固定容量的日志。

**理由**:
- 预先分配内存,无动态分配
- 满时自动覆盖旧条目
- 无论日志量如何,内存占用恒定
- O(1) 追加和查找操作

**考虑的替代方案**:
- **数组切片**: 当前方法,修剪时创建垃圾
- **LRU 缓存**: 更复杂,为不同用例设计
- **对象池**: 有帮助但不能解决数组调整大小问题

### 3. 批量处理传入消息

**决策**: 在缓冲区中累积消息,每 500ms 批量处理而不是立即处理。

**理由**:
- 将状态更新从每秒约 3-4 次减少到 2 次
- 将渲染成本分摊到多个日志条目
- 允许 requestAnimationFrame 批量 DOM 更新
- 用户不会注意到日志显示的 500ms 延迟

**考虑的替代方案**:
- **RequestIdleCallback**: 时机不可靠,可能延迟日志太多
- **立即处理**: 当前方法,导致高 GC 频率

### 4. 使用稳定键记忆化渲染函数

**决策**: 为处理的日志条目使用 `useMemo`,为 Virtuoso 使用稳定键。

**理由**:
- 防止每次渲染时重新创建日志条目对象
- React 可以重用 fiber 节点而不是创建新节点
- 稳定键使 virtuoso 能够高效重用 DOM

**当前问题**:
- 每次渲染时重新创建 `renderItemContent`(第 27 行)
- 某些情况下使用数组索引而不是稳定 ID 作为键
- 没有处理过的日志内容的记忆化

### 5. 为日志条目实现对象池

**决策**: 创建可重用日志条目对象池,而不是分配新对象。

**理由**:
- 减少对象创建的 GC 压力
- 池大小等于最大行数(一次分配 20k 个对象)
- 当它们滚动出视图时重用对象

**考虑的替代方案**:
- **无池化**: 当前方法,随时间创建数百万个对象
- **WeakMap**: 对创建率没有帮助

## 风险 / 权衡

### 风险 1: 可变 Ref 使调试更难
**风险**: 使用 `useRef` 而不是 state 使 React DevTools 中的调试更难。
**缓解**: 添加使用 state 的调试模式,在生产环境中谨慎使用 `console.log`。

### 风险 2: 500ms 批量延迟可能感觉迟缓
**风险**: 用户可能注意到日志显示有轻微延迟。
**缓解**:
- 保持批量延迟可配置(默认 500ms)
- 在突发期间显示"正在接收日志..."指示器
- 用户在底部时优先立即刷新

### 风险 3: 循环缓冲区丢失旧日志
**风险**: 缓冲区满时,最旧的日志被覆盖(与当前切片行为相同)。
**缓解**: 这是预期行为,无需更改。

### 风险 4: 复杂实现可能引入错误
**风险**: 手动内存管理容易出错。
**缓解**:
- 添加全面的单元测试
- CI 中的内存分析
- 使用功能标志逐步推出

## 迁移计划

### 阶段 1: 添加批量处理(低风险)
1. 添加消息缓冲区以累积传入日志
2. 添加定时器每 500ms 刷新缓冲区
3. 保留现有 `dealMessageData` 逻辑
4. 过夜测试,测量内存改进

### 阶段 2: 实现循环缓冲区(中风险)
1. 创建具有循环缓冲区实现的 `LogBuffer` 类
2. 用 `LogBuffer` 实例替换 `logAllDataRef`
3. 更新 `dealMessageData` 使用缓冲区
4. 添加内存分析测试

### 阶段 3: 优化渲染(低风险)
1. 记忆化 `renderItemContent`
2. 为所有日志条目使用稳定键
3. 为处理的日志内容添加 `useMemo`
4. 用 20k 条目测试,测量渲染性能

### 回滚策略
- 每个阶段可以独立回滚
- 功能标志在旧/新实现之间切换
- 部署后保留旧代码路径 2 周

## 待解决问题

1. 我们是否应该在设置中使批量延迟可配置?
   - **建议**: 是,默认 500ms,允许 100ms-2000ms 范围

2. 我们是否需要在组件重新挂载时保留日志?
   - **建议**: 否,卸载时清理以释放内存

3. 我们是否应该向 UI 添加内存使用指示器?
   - **建议**: 是,在工具提示中显示:"使用 X MB 存储 Y 条日志"

4. 如何处理极端突发(1 秒内 1000+ 日志)?
   - **建议**: 限制缓冲区大小,丢弃最旧的,显示警告
