## 新增需求

### 需求: 日志的循环缓冲区存储
系统应使用循环缓冲区(环形缓冲区)存储日志条目,具有固定的最大容量(20,000 条条目)。

#### 场景: 缓冲区接受新日志
- **当**新日志条目到达时
- **则**系统应以 O(1) 时间将其追加到缓冲区
- **且**如果缓冲区已满,最旧的条目应被覆盖

#### 场景: 缓冲区保持固定内存占用
- **当**连续添加日志超过 24 小时时
- **则**内存使用不应超过缓冲区容量(20k 条目约 50MB)
- **且**内存不应随日志量线性增长

### 需求: 自动清理旧日志条目
系统应自动移除超过最大容量的日志条目,无需手动干预。

#### 场景: 强制执行容量限制
- **当**缓冲区达到 20,000 条条目时
- **则**最旧的条目应被移除
- **且**总条目不应超过 20,000

#### 场景: 长时间运行无内存泄漏
- **当**日志监控器在连续日志流下运行 8 小时以上时
- **则**内存使用应稳定在一致水平
- **且**GC 频率应比基于数组的存储减少

### 需求: 高效的日志条目重用
系统应在日志条目滚动出视图时重用它们,以减少对象分配。

#### 场景: 日志条目的对象池
- **当**日志条目滚动出可见视口时
- **则**系统应将对象返回到池中以供重用
- **且**当池中有可用项时,系统不应为后续条目分配新对象

#### 场景: 池大小匹配缓冲区容量
- **当**系统初始化对象池时
- **则**池大小应至少等于最大缓冲区容量
- **且**正常操作期间不应发生额外分配

### 需求: 内存使用监控
系统应提供内存使用的可见性以供调试。

#### 场景: 内存使用指示器
- **当**用户悬停在日志计数指示器上时
- **则**系统应以 MB 为单位显示当前内存使用
- **且**显示缓冲区中的日志条目数量

#### 场景: 高内存的控制台警告
- **当**内存使用超过 100MB 时
- **则**系统应向控制台记录警告
- **且**建议清除日志以释放内存
