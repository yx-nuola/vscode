## 新增需求

### 需求: 处理前的消息累积
系统应在处理日志消息之前将其累积在缓冲区中,然后批量处理。

#### 场景: 缓冲区累积
- **当**日志消息以 300ms 间隔到达时
- **则**系统应在临时缓冲区中累积消息
- **且**不应立即触发 React 状态更新

#### 场景: 定时器上的批量刷新
- **当**自上次批量刷新以来已过去 500ms 时
- **则**系统应在单个批次中处理所有累积的消息
- **且**触发单个状态更新

### 需求: 可配置的批量延迟
系统应允许配置批量处理延迟。

#### 场景: 默认批量延迟
- **当**未配置自定义延迟时
- **则**系统应使用 500ms 作为默认批量延迟

#### 场景: 自定义批量延迟
- **当**配置了自定义延迟(100ms 到 2000ms 范围)时
- **则**系统应使用配置的延迟进行批量处理
- **且**应以警告拒绝超出允许范围的值

### 需求: 关键场景的立即刷新
系统应在某些条件下立即刷新批量缓冲区。

#### 场景: 用户滚动到底部时刷新
- **当**用户锁定在日志视图底部时
- **则**系统应优先立即显示新日志
- **且**将批量延迟减少到 100ms 以加快更新

#### 场景: 组件卸载时刷新
- **当**组件即将卸载时
- **则**系统应刷新任何剩余的缓冲消息
- **且**清除缓冲区以防止内存泄漏

### 需求: 批量大小限制
系统应限制最大批量大小以防止 UI 冻结。

#### 场景: 最大批量大小
- **当**缓冲区中累积超过 1000 条消息时
- **则**系统应立即处理批次
- **且**记录突发流量警告

#### 场景: 突发处理
- **当**1 秒内到达 1000+ 条日志时
- **则**系统应限制缓冲区并丢弃最旧的消息
- **且**向用户显示警告指示器

### 需求: 高效的批量处理
系统应优化批量消息处理以最小化对象创建。

#### 场景: 最小化对象创建
- **当**处理一批消息时
- **则**系统应尽可能重用现有对象
- **且**应避免创建不必要的中间数组

#### 场景: 每批次单次状态更新
- **当**处理批次时
- **则**系统应执行恰好一次状态更新
- **且**不触发中间重新渲染
